## Overview

Mikralink is a full-stack URL shortening service built with production engineering patterns. It goes beyond the typical "URL shortener tutorial" by implementing multi-tenancy, Redis caching, async analytics, per-plan rate limiting, and a dual-theme frontend — the same architectural decisions used by services like Bitly and Rebrandly.

The goal was to build something a senior engineer would be comfortable putting into production, not just a demo that works once.

## Technical Design Decisions

### Base62 Short Codes

Short codes are generated by Base62-encoding the database auto-increment ID, not by hashing or random generation.

```python
ALPHABET = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789"

def encode(num: int) -> str:
    result = []
    while num > 0:
        result.append(ALPHABET[num % 62])
        num //= 62
    return "".join(reversed(result))
```

**Why:** Uniqueness is guaranteed by the database — no collision checking or retry loops needed. ID 1,000,000 encodes to 4 characters. 7 characters covers 62⁷ = 3.5 trillion URLs.

---

### Redis-First Redirect Path

Every redirect hits Redis before touching PostgreSQL. The cache is warmed at write time so the first request after creation is already fast.

```
GET /{code}
  │
  ├─ Redis HIT  → 302 redirect (sub-millisecond)
  │
  └─ Redis MISS → PostgreSQL lookup
                   │
                   ├─ Not found / inactive / expired → 4xx
                   │
                   └─ Found → warm cache → 302 redirect
                              └─ asyncio.create_task(record_click(...))
                                 Analytics written in background.
                                 User is already redirected.
```

302 is used instead of 301. A 301 is cached permanently by browsers — if a link is later deactivated, users who visited before would never see the deactivation. 302 forces a fresh lookup on every visit.

---

### API Key Authentication

Keys are generated with `secrets.token_hex(24)` (cryptographically secure, 48 hex chars), prefixed with `sk_live_`, and stored as SHA-256 hashes. The raw key is shown once and discarded — the same pattern used by Stripe and Twilio.

```python
raw_key   = f"sk_live_{secrets.token_hex(24)}"
key_hash  = hashlib.sha256(raw_key.encode()).hexdigest()
# Only key_hash is stored. raw_key is returned once and never saved.
```

**SQL bug fixed during audit:** The original WHERE clause had an operator precedence error that was a cross-tenant authentication bypass:

```sql
-- BUG: AND binds before OR — any non-expired key matched any tenant
WHERE key_hash = $1 AND expires_at IS NULL OR expires_at > NOW()

-- FIX: explicit parentheses
WHERE key_hash = $1 AND (expires_at IS NULL OR expires_at > NOW())
```

---

### Rate Limiting (Pipeline Atomic)

Rate limiting uses a Redis fixed-window counter with a critical race condition fix. The naive approach — `INCR` then `EXPIRE` as two separate commands — leaves the key with no TTL if the process dies between them, permanently locking the tenant.

```python
async with redis.pipeline(transaction=False) as pipe:
    pipe.incr(key)
    pipe.expire(key, 60, nx=True)  # nx=True: only set TTL if not already set
    count, _ = await pipe.execute()
return count <= max_requests
```

Both commands are sent in one round-trip. `nx=True` on EXPIRE means the window always resets 60 seconds after the first request, not after the most recent one — which is correct fixed-window behaviour.

---

### Guest Usage — UUID Token, Not IP

Guest (unauthenticated) usage is limited to 5 shortens per session. Early versions keyed this counter by IP address. That approach breaks in two ways:

1. **Incognito windows share the same IP** — a user opening a private window would hit the same exhausted counter.
2. **Shared IPs** — users behind the same office NAT or mobile carrier would all share one 5-use bucket.

The fix: the frontend generates a UUID using `crypto.randomUUID()` on first visit, stores it in `localStorage`, and sends it as an `X-Guest-Token` header. Each browser profile and each incognito session gets an independent counter.

```javascript
function getGuestToken() {
  let t = localStorage.getItem('mikralink_guest_token');
  if (!t) {
    t = crypto.randomUUID();
    localStorage.setItem('mikralink_guest_token', t);
  }
  return t;
}
```

The server sanitises the token before using it as a Redis key — only alphanumerics and hyphens are allowed, max 64 characters.

### Async Click Tracking

Analytics writes never block the redirect response. The user is sent to their destination immediately; the click record is written in a background task.

```python
response = RedirectResponse(url=original_url, status_code=302)

asyncio.create_task(
    record_click(
        short_code=short_code,
        ip_address=ip,
        user_agent=request.headers.get("user-agent"),
        referrer=request.headers.get("referer"),
    )
)

return response  # already returned — user is redirected
```

Click data is stored in a separate `clicks` table rather than a `click_count` column on `urls`. Concurrent INSERTs to a separate table are significantly faster than concurrent UPDATEs to the same row under high traffic.

---

### Multi-Tenancy & Plan Limits

Every URL is scoped to a tenant. Plan limits are stored in a `plan_limits` table rather than hardcoded, making them adjustable without a deployment.

| Plan       | URL Limit | Requests/min | Analytics Retention |
|------------|-----------|--------------|---------------------|
| free       | 100       | 10           | 30 days             |
| pro        | 10,000    | 100          | 365 days            |
| enterprise | unlimited | 1,000        | unlimited           |

## Database Schema

```sql
urls       — short_code (Base62), original_url, tenant_id, expires_at, is_active
clicks     — short_code (FK), ip_address (INET), user_agent, referrer, clicked_at
tenants    — name, email, plan, is_active
api_keys   — tenant_id (FK), key_hash (SHA-256), key_prefix, label, last_used
plan_limits— plan, max_urls, requests_per_minute, analytics_retention
```

Key indexes:
- `idx_urls_short_code` — primary redirect lookup path
- `idx_urls_tenant_active (tenant_id, is_active)` — composite, covers URL quota checks
- `idx_clicks_short_code` — analytics aggregation
- `idx_api_keys_hash` — auth lookup on every authenticated request

## API Reference

### Register

```bash
curl -X POST https://url-shortner-4okq.onrender.com/tenants/register \
  -H "Content-Type: application/json" \
  -d '{"name": "Your Name", "email": "you@example.com"}'

# → { "api_key": "sk_live_...", "warning": "Save this key — shown once only." }
```

### Shorten a URL

```bash
curl -X POST https://url-shortner-4okq.onrender.com/shorten \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com/very/long/path"}'

# → { "short_code": "3Kp", "short_url": "https://url-shortner-4okq.onrender.com/3Kp" }
```

### View Analytics

```bash
curl https://url-shortner-4okq.onrender.com/urls/3Kp/stats \
  -H "Authorization: Bearer sk_live_..."

# → { "total_clicks": 42, "clicks_by_day": [...], "top_referrers": [...] }
```

### Guest Shorten (no auth, 5 uses per session)

```bash
curl -X POST https://url-shortner-4okq.onrender.com/shorten/guest \
  -H "Content-Type: application/json" \
  -H "X-Guest-Token: <uuid>" \
  -d '{"url": "https://example.com"}'
```

Full interactive docs available at `/docs`.

## Deployment

The service runs on Render (Docker runtime) with PostgreSQL on Neon and Redis on Upstash. All three have free tiers sufficient for portfolio-scale traffic.

Required environment variables on Render:

| Variable       | Description                                      |
|----------------|--------------------------------------------------|
| `DATABASE_URL` | Neon PostgreSQL connection string                |
| `REDIS_URL`    | Upstash Redis URL (`rediss://...`)               |
| `BASE_URL`     | Public URL of this service (for short_url field) |
| `ENV`          | Set to `production`                              |

Render auto-deploys on every push to `main`. The `render.yaml` in the repo root configures the service type, plan, and health check path (`/health`).
